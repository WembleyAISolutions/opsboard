# OpsBoard Signal Protocol v2

## 1. Overview

OpsBoard Signal Protocol v2 defines a workflow-aware extension of the signal protocol used by OpsBoard.

### Current state (v1)

v1 is a signal and notification routing protocol. It standardizes event emission, validation, and normalization so signals can be surfaced in modules such as Inbox, Signal, and Approvals.

### Future state (v2)

v2 extends v1 to support workflow tracking metadata and controlled execution hints. The goal is to move OpsBoard from a signal board toward an AI Workflow OS control layer while preserving existing signal behavior.

v2 is a protocol-level evolution. It does not imply that all workflow semantics are already implemented in engine or UI.

## 2. Design Goals

v2 is designed to support:

- workflow tracking
- approval tracking
- routing guidance
- controlled auto execution
- backward-compatible evolution from v1

## 3. Base v1 Fields

The following fields remain core in v2:

- `protocol_version`: protocol contract version string.
- `signal_id`: unique identifier for a signal event.
- `source_ai`: emitting AI source identity.
- `signal_kind`: semantic event type (input, attention, approval, etc.).
- `signal_priority`: urgency level for ordering.
- `human_action_needed`: whether explicit operator action is required.
- `title`: short human-readable label.
- `summary`: concise human-readable context.
- `timestamp`: ISO 8601 event time.

These fields preserve v1 interoperability and remain mandatory for producer compatibility.

## 4. New v2 Fields

### `workflow_id`

- **Definition**: stable identifier shared by signals in the same end-to-end workflow.
- **Purpose**: enables cross-signal grouping for one business/home/dev/tax process.
- **When to use**: whenever multiple signals belong to one evolving matter.
- **Notes / constraints**:
  - should remain stable through the workflow lifecycle
  - should not be reused across unrelated workflows

### `approval_id`

- **Definition**: identifier for a specific approval decision context.
- **Purpose**: links request, decision, and follow-up signals to one yes/no chain.
- **When to use**: signals requiring explicit approval tracking.
- **Notes / constraints**:
  - may differ from `workflow_id`; one workflow can include multiple approvals
  - should be generated by the coordinating producer (often Admin AI)

### `routing_hint`

- **Definition**: producer-provided suggested destination/classification.
- **Purpose**: helps downstream routing without forcing behavior.
- **When to use**: when producer has domain context on where operator attention should go.
- **Notes / constraints**:
  - advisory only; engine/policy remains final authority
  - must not be treated as executable command

### `auto_execute`

- **Definition**: producer intent flag indicating low-risk automatic continuation is acceptable.
- **Purpose**: supports future controlled automation evaluation.
- **When to use**: low-risk operations where producer proposes automatic execution.
- **Notes / constraints**:
  - default should be `false`
  - cannot by itself authorize execution
  - policy layer must evaluate risk and permission before any action

## 5. Recommended Schema Example

```json
{
  "protocol_version": "2.0",
  "signal_id": "finance-2026-03-13-001",
  "source_ai": "FINANCE_AI",
  "signal_kind": "approval",
  "signal_priority": "high",
  "human_action_needed": true,
  "title": "Invoice approval required",
  "summary": "Supplier invoice requires approval before payment scheduling.",
  "timestamp": "2026-03-13T10:20:00.000Z",
  "workflow_id": "wf-finance-mar-2026-01",
  "approval_id": "ap-invoice-4200-aud-01",
  "routing_hint": "approval_queue",
  "auto_execute": false
}
```

## 6. Field Semantics

- `workflow_id` is used to track one matter across multiple signals.
- `approval_id` is used for yes/no decision tracking in approval chains.
- `routing_hint` is a routing suggestion, not a mandatory instruction.
- `auto_execute` does not independently authorize execution.

## 8. Safety Model

- `auto_execute` should default to `false`.
- Money, tax, production deploy, and legal-compliance scenarios must not auto-execute from signal alone.
- Even with `auto_execute=true`, a policy layer must evaluate risk before execution.
- A signal is not direct execution authorization.

## 10. Admin Hub Usage Model

In a multi-AI operating model, Admin AI (or equivalent coordinator) should:

- assign `workflow_id` to related signals from all participating AIs
- assign `approval_id` for each distinct decision checkpoint
- set `routing_hint` to propose queue destination
- enforce `auto_execute=false` for high-risk items

This provides consistent coordination while preserving engine-side control.

## 11. Migration Guidance

v1 to v2 should be incremental:

- v1 producers continue to work unchanged.
- v2 fields can be introduced as optional extension fields first.
- bridge, hook, and producers can adopt new fields gradually.
- engine and UI can later add workflow semantics in stages.

This avoids breaking active operations while enabling protocol growth.

## 12. Recommended Implementation Phases

- **Phase 1**: document only
- **Phase 2**: producer support
- **Phase 3**: bridge + hook support
- **Phase 4**: UI support
- **Phase 5**: engine workflow grouping
- **Phase 6**: controlled automation

## 13. Final Positioning

- v1 = AI signal router
- v2 = AI workflow control protocol
- OpsBoard can evolve toward AI Workflow OS

v2 should be treated as protocol foundation for future workflow-aware behavior, while current runtime may still operate primarily with v1-compatible routing semantics.
